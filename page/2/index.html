代理模式

什么是代理

代理(proxy) 意思就是不用你自己去做，别人帮你做了,
这个就是代理。。

静态代理

有一个接口

一个实现类

这时，如果我想在这个方法前后加一些逻辑处理，该怎么做呢？
如果直接写死在方法里面肯定会被架构师骂的够呛的O(∩_∩)O…

这时，我们可以这样做
写一个代理类,在调用方法之前后分别可以做逻辑操作

TankProxy也实现Tank接口，并在构造里面new出TankImpl是为了可以在TankProxy调用TankImpl的tototo方法
然后在调用方法之前可以做一些逻辑的操作，这个就是代理啦
客户端调用就很简单了，去试试效果。。。

静态代理每一次的代理都会产生一个代理类，
当一个工程庞大时，用静态代理显然就不合适了，（会造成类过多）

动态代理

使用静态代理会产生类过多的问题，所以就要用动态代理啦
动态代理其实就是在程序运行时，运用反射机制动态创建代理对象

使用动态代理首先是要实现java.lang.reflect.InvocationHandler这个接口

其中，定义了一个target变量，它就是被代理的对象，通过构造来初始化
invoke方法是InvocationHandler这个接口里面的方法，必须实现，其中
Object proxy：指被代理的对象。
Method method：要调用的方法
Object[] args：方法调用时所需要的参数

我还新增了一个getProxy方法,利用jdk提供给我们的Proxy类的工厂方法newProxyInstance方法得到代理对象
newProxyInstance方法参数：

  • 参数1：ClassLoader
  • 参数2：该实现类所有接口，包括其祖宗的…
  • 参数3：动态代理对象

CGLib代理

总是会有解决办法的，嘎嘎，就是这个CGLib代理咯
CGLib是一个在运行期间动态生成字节码的工具，也就是动态产生代理类。。
来写一个CGlibProxy，会依赖cglib的jar到时自己去下载一份就ok了

要用CGLib代理，需要实现net.sf.cglib.proxy.MethodInterceptor这个接口，并填充intercept方法，
最后一个参数methodProxy，CGLib给我们提供的是方法级别的代理。。

dubbo理解

dubbo官网

背景

随着互联网的发展，网站应用的规模不断扩大，常规的垂直应用架构已无法应对，
分布式服务架构以及流动计算架构势在必行，亟需一个治理系统确保架构有条不紊的演进

dubbo架构

在这里只描述一下他们的说明，dubbo官网有架构图。

根据官方的架构图以及解释说明可以看出

  • Provider:顾名思义，供养者，提供者，即为服务提供方
  • Consumer：消费服务的一方
  • Registry：服务注册于发现的注册中心
  • Monitor：监视器，统计服务的调用次数及时间
  • Container：容器，服务运行的容器

角色关系

1.Container负责启动，加载，运行服务提供者
2.Provider启动时向Registry注册自己的服务
3.Consumer启动时，向Registry订阅自己所需的服务
4.Container返回Provider地址列表给Consumer，如果地址有变更，
  Container将基于长连接推送变更给Consumer
5.Consumer，从Provider地址列表中，基于软负载均衡算法，选一台Provider进行调用，失败则选另一台
6.Provider,Consumer,在内存中累计调用次数和调用时间，定时每分钟发送一次统计数据到Monitor
